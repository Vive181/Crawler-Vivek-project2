# Build Log – Session 2

**Date:** July 9
**Duration:** 120 minutes

## Goal

Improve the reliability of the crawler by adding retry support to the URL Frontier and investigate a method for distinguishing static HTML pages from JavaScript framework-based pages.

## Problem

When a URL fetch fails due to temporary issues such as network timeouts or server errors, the crawler should not immediately discard the URL. Without a retry mechanism, temporary failures could result in missing pages.

Another challenge was determining whether a webpage is a static HTML page or generated by a JavaScript framework before introducing browser rendering.

## Changes Made

### Retry Count in `URLNode`

**Purpose:**
Added a `retryCount` field to the `URLNode` structure.

**Why:**
To keep track of the number of failed fetch attempts and allow temporary failures to be retried before permanently discarding the URL.

---

### Retry Workflow

**Purpose:**
Designed a retry mechanism for failed fetch requests.

**Why:**
The workflow allows a URL to be reinserted into the frontier if its retry count is below the configured maximum limit, improving crawler reliability while preventing infinite retry loops.

---

### Initial Framework Detection Strategy

**Purpose:**
Explored a threshold-based approach for identifying JavaScript framework-generated pages.

**Why:**
The idea was to distinguish static HTML pages from dynamically rendered pages before deciding whether browser rendering would be required.

## What I Tried

Implemented retry count support inside the `URLFrontier` data structure and designed the retry workflow for handling temporary failures. I also investigated using a threshold-based scoring system based on indicators such as JavaScript bundles, framework-specific attributes, script size, and minimal HTML content to classify webpages.

## Issue Encountered

The threshold-based approach introduced a design limitation. Since the crawler begins with a **seed URL**, there is no initial baseline for calculating a reliable threshold value. Different websites also have significantly different structures, making a fixed threshold unsuitable for large-scale crawling. As a result, this approach could incorrectly classify webpages during the early stages of crawling.

## Outcome

Successfully added retry count support to the `URLNode` structure and designed a retry workflow for future failure recovery. After evaluating the threshold-based framework detection approach, I concluded that it is not reliable as the primary detection mechanism. The decision was made to complete the Fetcher first and implement framework detection later as a dedicated module that can integrate with browser rendering when needed.
